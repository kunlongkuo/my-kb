# LLM Wiki (LLM 維基)

一個使用 LLM 建立個人知識庫的模式。

這是一個概念文件，旨在供您複製並貼上到您自己的 LLM 代理程式（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。其目標是傳達高層次的想法，而您的代理程式將與您合作建立具體細節。

## 核心理念

大多數人使用 LLM 和文件的經驗就像 RAG（檢索增強生成）：您上傳一堆文件，LLM 在查詢時檢索相關片段並生成答案。這雖然有效，但 LLM 每次都在從頭開始重新發現知識，沒有積累。問一個需要綜合五個文件的微妙問題，LLM 每次都必須尋找並拼湊相關片段。沒有任何東西被建立起來。NotebookLM、ChatGPT 文件上傳和大多數 RAG 系統都是這樣運作的。

這裡的想法不同。LLM 不僅僅是在查詢時從原始文件中檢索，而是**增量地建立並維護一個持久的維基 (wiki)** —— 一個位於您與原始來源之間、結構化且相互連結的 markdown 文件集合。當您添加新來源時，LLM 不僅僅是將其索引以供以後檢索。它會閱讀它、提取關鍵資訊，並將其整合到現有的維基中 —— 更新實體頁面、修正主題摘要、標註新數據與舊主張矛盾之處，加強或挑戰不斷演變的綜合。知識被編譯一次，然後 *保持更新*，而不是在每次查詢時重新衍生。

這是關鍵區別：**維基是一個持久、具有複利效應的產物。** 交叉引用已經存在。矛盾之處已經被標記。綜合已經反映了您所讀過的一切。隨著您添加的每個來源和提出的每個問題，維基會變得越來越豐富。

您（或很少）親自編寫維基 —— LLM 編寫並維護其所有內容。您負責尋找來源、探索並提出正確的問題。LLM 負責所有的瑣事 —— 摘要、交叉引用、歸檔和簿記，這些工作讓知識庫隨時間變得真正有用。在實踐中，我一側開啟 LLM 代理程式，另一側開啟 Obsidian。LLM 根據我們的對話進行編輯，我則實時瀏覽結果 —— 跟隨連結、檢查圖譜視圖、閱讀更新的頁面。Obsidian 是 IDE；LLM 是程式設計師；維基是程式碼庫。

這可以應用於許多不同的情境。舉幾個例子：

- **個人**：追蹤您自己的目標、健康、心理學、自我提升 —— 歸檔日記條目、文章、播客筆記，並隨時間建立起關於您自己的結構化圖譜。
- **研究**：在數週或數月內深入探討一個主題 —— 閱讀論文、文章、報告，並增量地建立一個具有演進論點的綜合維基。
- **閱讀書籍**：隨著進度歸檔每一章，為角色、主題、情節線索以及它們如何連接建立頁面。到最後，您就有了一個豐富的伴侶維基。想像一下像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 這樣的粉絲維基 —— 數千個相互關聯的頁面，涵蓋了由志願者社群歷經多年建立的角色、地點、事件、語言。您可以在閱讀時親自建立類似的東西，由 LLM 處理所有的交叉引用和維護。
- **商務/團隊**：由 LLM 維護的內部維基，餵入 Slack 討論串、會議紀錄、專案文件、客戶電話。可能由人類參與審核更新。維基保持更新，因為 LLM 承擔了團隊中沒人想做的維護工作。
- **競爭分析、盡職調查、旅遊規劃、課堂筆記、愛好深挖** —— 任何您隨時間積累知識並希望其有組織而非分散的領域。

## 架構

分為三層：

**原始來源 (Raw sources)** —— 您策劃的源文件集合。文章、論文、圖像、數據文件。這些是不可變的 —— LLM 讀取它們但從不修改它們。這是您的真實來源 (source of truth)。

**維基 (The wiki)** —— 一個由 LLM 生成的 markdown 文件目錄。摘要、實體頁面、概念頁面、比較、概覽、綜合。LLM 完全擁有這一層。它建立頁面，在新來源到達時更新它們，維護交叉引用，並保持一切一致。您閱讀它；LLM 編寫它。

**架構 (The schema)** —— 一個告訴 LLM 維基如何結構化、有哪些慣例，以及在攝取來源、回答問題或維護維基時要遵循哪些工作流程的文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）。這是關鍵的設定檔 —— 它使 LLM 成為一個有紀律的維基維護者，而不是一個通用的聊天機器人。隨著您弄清楚什麼適合您的領域，您和 LLM 會隨時間共同演進它。

## 操作

**攝取 (Ingest)**。您將新來源放入原始集合中，並告訴 LLM 處理它。一個範例流程：LLM 閱讀來源，與您討論關鍵要點，在維基中編寫摘要頁面，更新索引，更新整個維基中相關的實體和概念頁面，並在日誌中追加一條條目。單個來源可能會觸動 10-15 個維基頁面。就我個人而言，我偏好一次攝取一個來源並保持參與 —— 我閱讀摘要、檢查更新，並指導 LLM 要強調什麼。但您也可以在較少監督的情況下批量攝取多個來源。由您來開發適合您風格的工作流程，並將其記錄在架構文件中供未來會話使用。

**查詢 (Query)**。您針對維基提問。LLM 搜尋相關頁面，閱讀它們，並綜合出帶有引用的答案。答案可以根據問題採取不同的形式 —— markdown 頁面、比較表、簡報檔 (Marp)、圖表 (matplotlib)、畫布 (canvas)。重要的洞察：**好的答案可以作為新頁面存回維基。** 您要求的比較、分析、您發現的連結 —— 這些都是有價值的，不應該消失在聊天紀錄中。這樣一來，您的探索就像攝取的來源一樣，在知識庫中產生複利。

**檢查 (Lint)**。定期要求 LLM 對維基進行健康檢查。尋找：頁面之間的矛盾、已被新來源取代的陳舊主張、沒有傳入連結的孤兒頁面、提到但缺乏專屬頁面的重要概念、缺失的交叉引用、可以透過網路搜尋填補的數據空白。LLM 擅長建議要調查的新問題和要尋找的新來源。這能保持維基在成長過程中的健康。

## 索引與日誌

兩個特殊文件幫助 LLM（以及您）在維基成長時進行導航。它們有不同的用途：

**index.md** 是以內容為導向的。它是維基中所有內容的目錄 —— 每個頁面都列有連結、一行摘要，以及可選的元數據（如日期或來源計數）。按類別（實體、概念、來源等）組織。LLM 在每次攝取時都會更新它。在回答查詢時，LLM 首先閱讀索引以尋找相關頁面，然後深入研究它們。這在中等規模（~100 個來源，~數百個頁面）下運作得非常好，且不需要基於嵌入 (embedding) 的 RAG 基礎設施。

**log.md** 是按時間順序排列的。它是發生了什麼事以及何時發生的追加記錄 —— 攝取、查詢、健康檢查。一個有用的技巧：如果每個條目都以一致的前綴開頭（例如 `## [2026-04-02] ingest | 文章標題`），那麼該日誌就可以使用簡單的 unix 工具進行解析 —— `grep "^## \[" log.md | tail -5` 可以顯示最後 5 條條目。日誌為您提供了維基演進的時間線，並幫助 LLM 了解最近做了什麼。

## 選配：CLI 工具

在某些時候，您可能想要建立一些小工具來協助 LLM 更有效地操作維基。對維基頁面進行搜尋的引擎是最顯而易見的 —— 在小規模下，索引文件就足夠了，但隨著維基的成長，您會需要真正的搜尋功能。[qmd](https://github.com/tobi/qmd) 是一個不錯的選擇：它是一個用於 markdown 文件的本地搜尋引擎，具有混合 BM25/向量搜尋和 LLM 重新排名功能，全部在裝置上運行。它既有 CLI（以便 LLM 可以調用它），也有 MCP 伺服器（以便 LLM 可以將其用作原生工具）。您也可以自己建立更簡單的東西 —— LLM 可以協助您在需要時「感性開發」(vibe-code) 一個簡易的搜尋腳本。

## 提示與技巧

- **Obsidian Web Clipper** 是一個瀏覽器擴充功能，可以將網頁文章轉換為 markdown。對於快速將來源放入您的原始集合非常有用。
- **本地下載圖像**。在 Obsidian 設定 → 檔案與連結中，將「附件資料夾路徑」設定為固定目錄（例如 `raw/assets/`）。然後在設定 → 快捷鍵中，搜尋「下載」以找到「為目前檔案下載附件」並將其綁定到快捷鍵（例如 Ctrl+Shift+D）。剪輯文章後，按下快捷鍵，所有圖像都會下載到本地硬碟。這是選配但很有用 —— 它讓 LLM 可以直接查看和引用圖像，而不是依賴可能失效的 URL。請注意，LLM 無法原生在一次讀取中處理包含內嵌圖像的 markdown —— 解決方法是讓 LLM 先閱讀文本，然後單獨查看部分或全部引用的圖像以獲得額外的背景資訊。這有點笨拙，但效果不錯。
- **Obsidian 的圖譜視圖 (graph view)** 是查看維基形狀的最佳方式 —— 誰連接到誰、哪些頁面是中心樞紐、哪些是孤兒。
- **Marp** 是一種基於 markdown 的簡報格式。Obsidian 有相關的外掛程式。適用於直接從維基內容生成簡報。
- **Dataview** 是一個 Obsidian 外掛程式，可在頁面 frontmatter 上執行查詢。如果您的 LLM 將 YAML frontmatter 添加到維基頁面（標籤、日期、來源計數），Dataview 可以生成動態表格 and 列表。
- 維基就只是一個 markdown 文件的 git 倉庫。您可以免費獲得版本歷史、分支和協作功能。

## 為什麼這有效

維護知識庫最乏味的部分不是閱讀或思考 —— 而是簿記。更新交叉引用、保持摘要更新、標註新數據何時與舊主張矛盾、保持數十個頁面的一致性。人類放棄維基是因為維護負擔的增長速度超過了價值。LLM 不會感到無聊，不會忘記更新交叉引用，並且可以在一次處理中觸動 15 個檔案。維基得以維持，是因為維護成本接近於零。

人類的工作是策劃來源、引導分析、提出好問題，並思考這一切意味著什麼。其餘的一切都是 LLM 的工作。

這個想法在精神上與 Vannevar Bush 的 Memex (1945) 相關 —— 一個個人化、策劃過的知識庫，在文件之間具有關聯路徑。Bush 的願景與此更為接近，而不是後來的全球資訊網：私密的、主動策劃的，文件之間的連接與文件本身一樣有價值。他無法解決的部分是誰來負責維護。LLM 處理了這個問題。

## 備註

這份文件刻意保持抽象。它描述的是想法，而非特定的實踐。確切的目錄結構、架構慣例、頁面格式、工具 —— 所有這些都將取決於您的領域、您的偏好和您選擇的 LLM。上面提到的所有內容都是選配且模組化的 —— 挑選有用的，忽略沒用的。例如：您的來源可能僅限文本，因此您根本不需要圖像處理。您的維基可能足夠小，索引文件就是您所需要的，不需要搜尋引擎。您可能不在意簡報，只想使用 markdown 頁面。您可能想要一套完全不同的輸出格式。使用此文件的正確方法是將其共享給您的 LLM 代理程式，並共同協作實例化一個符合您需求的版本。這份文件的唯一任務是溝通模式。您的 LLM 可以弄清楚其餘的部分。
